# Опросник «Профиль туриста» — спецификация v2

_Автор содержания: владелец (Эдуард), 2026-07-05. Технические решения по хранению и интеграции — внизу. Канон для реализации опросника в Задании 12; дополняет `prospects-crm-spec.md`._

## Принципы

- Тон: короткие вопросы, пояснение «зачем спрашиваем» под каждым чувствительным вопросом. Пояснение — одна строка, серым, без восклицаний и продающих слов.
- Длина: 8–10 экранов в зависимости от ветки, прогресс-бар обязателен («вопрос 3 из 9»).
- Ни одного обязательного поля про деньги. Бюджетный вопрос сформулирован через комфорт, с опцией «покажите варианты — сориентируюсь».
- Один вопрос — один экран (мобильный формат), кроме блока «состав группы».
- Опросник — **отдельный лендинг**, который владелец пересылает ссылкой.
- Термин «золотой маршрут» и профессиональный жаргон в интерфейсе не используются — только названия городов.

## Логика ветвления

```
Q1 Даты
Q2 Первая поездка?
 ├── ДА  → Q3a Токио + Киото: оставить / добавить / поменять
 └── НЕТ → Q3b Где уже были (свободный текст)
            → Q3c Повторить классику или хотим новое?
               └── если «новое»/«микс» → Q3d Известные маршруты или редкая Япония?
Q4 Состав группы
Q5 Дети / мобильность
Q6 Специальные интересы
Q7 Ритм поездки
Q8 Диапазон стоимости отеля за ночь ($, ползунок)
Q9 Бронирование отелей
Q10 Формат сопровождения
 └── если «гид на весь маршрут» → Q10a Один гид или локальные гиды
Q11 Свободный комментарий + контакт
```

## Блоки

### Блок 1. Даты
«Когда планируете поездку?» — календарь-диапазон; переключатель `Даты точные` / `Примерные, ±несколько дней` / `Пока знаю только месяц` (тогда — месяц+год из списка).
Пояснение: «Цены отелей и доступность гидов в Японии сильно зависят от сезона — сакура, момидзи и праздники бронируются за полгода. Чем раньше знаем даты, тем лучше варианты.»

### Блок 2. Опыт
«Вы уже бывали в Японии?» — `Да` / `Нет, это первая поездка`. Без пояснения.

**Q3a (первая поездка):** «Регионы Токио и Киото — основной выбор для тех, кто приезжает в Японию в первый раз. Хотели бы что-то добавить или поменять?»
`Токио и Киото — то, что нужно` / `Хотели бы что-то добавить` (подсказка серым: чаще всего добавляют Осаку и Нару, Канадзаву, Хаконэ/Фудзи или Хиросиму + необязательное поле «что именно?») / `Хотели бы что-то поменять` (поле «расскажите, что вам ближе») / `Не знаем — доверимся вашей рекомендации`.
Пояснение: «Эти два региона покрывают главное и не превращают первую поездку в гонку. Добавлять имеет смысл, если у вас 12+ дней — подскажем по вашим датам.»
Дальше новичок вопросов про регионы не видит.

**Q3b (уже были):** «Где вы уже были в Японии? Напишите своими словами.» — свободный текст, плейсхолдер: «Например: Токио, Киото, Осака, Хиросима, Сендай…». Списка вариантов нет намеренно.

**Q3c:** «Как поступим с местами, где вы уже были?» — `Хотим только новое` / `Были давно или едем в новом составе — готовы повторить классический маршрут` / `Микс: знакомые города как базы, программа внутри — новая`.
Пояснение: «Вернуться в Токио и Киото — не шаг назад: с детьми или друзьями это фактически новая поездка. А для тех, кто хочет только новое, Япония глубже, чем кажется.»

**Q3d (если «только новое»/«микс»):** «Какого рода новое вам интересно?» — `Известная Япония, где мы ещё не были` (Канадзава, Хиросима, Хоккайдо, Кюсю…) / `Редкие направления и экзотика` / `И то и другое — соберите микс` + необязательное поле «Есть конкретные идеи?».
Пояснение: «Редкие направления — это другая логистика: меньше английского, реже транспорт, зато Япония без туристов. Честно предупредим о плюсах и минусах каждого варианта.»

### Блок 3. Состав группы
«Кто едет?» — степперы взрослые/дети (дети >0 → возраст каждого), переключатель `Состав окончательный` / `Может измениться`.
Пояснение: «От состава зависят номера в отелях (в Японии строгие лимиты по числу гостей в номере) и тип транспорта.»

### Блок 4. Мобильность
«Есть ли в группе участники, для которых важен щадящий темп?» — мультивыбор: `Дети до 6 лет` / `Участники старше 70` / `Ограниченная мобильность / сложно с лестницами` / `Нужен лифт / минимальные пешие переходы` / `Нет, все готовы много ходить`.
Пояснение: «В Японии много лестниц, переходов и станций без лифтов. Зная это заранее, мы строим маршрут так, чтобы никому не пришлось терпеть.»

### Блок 5. Специальные интересы
«Есть ли темы, вокруг которых хочется построить часть поездки — или всю её целиком?» — мультивыбор: `Гастрономия` / `Горные походы / хайкинг` / `Outdoor` / `Фотография` / `Охота на искусство` / `Ремёсла` / `Ничего специального` + свободное поле.
Если выбран интерес — переключатель: `Акцент внутри обычной поездки` / `Специализированный тур вокруг темы`.
Пояснение: «Один сильный интерес часто даёт лучший день всей поездки — ради него можно перестроить маршрут. Если таких тем нет, это тоже нормальный ответ.»

### Блок 6. Ритм
Три карточки: `Спокойный` (минимум переездов, 1–2 отеля, свободные вечера) / `Сбалансированный` (2–3 базы, насыщенно, без спешки) / `Интенсивный` (максимум, ранние выезды — и это в радость).
Пояснение: «Правильный ритм — главное, что отличает удачную поездку. Здесь нет правильного ответа, только ваш.»

### Блок 7. Стоимость отелей
«Укажите комфортный диапазон стоимости отеля за ночь.» — range-ползунок $80–$800+, шаг $20, значения над ручками. Без звёзд и категорий.
Чекбокс: `Хотя бы одну ночь — традиционный рёкан с онсэном`. Опция-выход: `Затрудняюсь — покажите варианты в разных категориях`.
Пояснение: «Спрашиваем не для того, чтобы предложить подороже. Наоборот — чтобы сразу показывать только то, что вам подходит, и не тратить ваше время на лишние варианты.»

### Блок 8. Бронирование отелей
`Бронирую сам(а) — нужен только маршрут` / `Дайте рекомендации, бронировать буду сам(а)` / `Возьмите бронирование на себя`.
Пояснение: «Любой вариант нормален. Если бронируете сами — просто дадим точные названия и лучшие даты для брони.»

### Блок 9. Формат сопровождения
Три карточки: `Самостоятельно, без гида` (детальный маршрут, логистика, поддержка на связи) / `Гид в отдельные дни` / `Гид на всём маршруте`.
Пояснение: «Формат влияет на стоимость сильнее всего остального, поэтому спрашиваем прямо. Комбинировать можно.»

**Q10a (если «гид на всём маршруте»):** `Один гид на всю поездку` / `Локальные гиды в каждом регионе` / `Как посоветуете`.
Пояснение: «Один гид — постоянство и комфорт, но его переезды и проживание входят в стоимость. Локальные гиды глубже знают свой регион и обычно выходят дешевле. Оба варианта рабочие.»

### Блок 10. Финал
«Что-то важное, о чём мы не спросили?» — свободное поле, подсказки: «аллергии, особые даты, страхи (землетрясения, языковой барьер)…»
Контакт: имя + канал (`Telegram` / `WhatsApp` / `Email`) + значение.
Финальный экран: «Спасибо! Мы изучим ответы и вернёмся с наброском маршрута в течение N дней. Это ни к чему вас не обязывает.»

## Продуктовые заметки владельца

- Финансовый вопрос стоит после ритма и интересов намеренно — к этому моменту доверие выше.
- Сохранение черновика («продолжить позже») снижает потери на длинной анкете.
- Q3a — мягкая демонстрация экспертизы без жаргона.

## Технические решения (Fable-5, согласуются с prospects-crm-spec.md)

**Хранение — JSON-канон + фильтруемые колонки.** Анкета богаче схемы Prospects и будет эволюционировать; миграции select-полей под каждую итерацию — тупик. Поэтому: полный payload пишется в `Fact Find Answers` (long text, JSON) — это канон для рендера «профиля туриста» в карточке; параллельно денормализуются колонки, нужные доске/дашборду/фильтрам: `Arrival Date`/`Departure Date`/`Flexible Dates` (из dates), `Party Size`/`Children` (из group), `Pace` (relaxed→relaxed, balanced→moderate, intense→active), `Mobility` (limited_mobility|elevator_needed→limited; иначе full), `Interests` (map: gastronomy→food, hiking→hiking, photography→photography, art→art; прочее живёт только в JSON), `Must See`/`Notes`, `Name`/`Contact`, новые `First Trip`, `Guide Format`, `Hotel Booking`, `Hotel Budget`.

**Два входа на лендинг:**
- `/profile/[token]` — персональная ссылка существующего prospect'а (создаётся при заявке с сайта; владелец пересылает из карточки). Submit обновляет prospect.
- `/profile` — общая ссылка без токена (пересылается кому угодно). Контакт собирается в финальном блоке → submit СОЗДАЁТ prospect (Source по umt-параметру `?src=telegram|social|...`, иначе website) с заполненным профилем.
Черновик — localStorage (v1), «продолжить позже» работает на том же устройстве.

**Карточка клиента:** «Профиль туриста» — компактный блок в самом верху (рендер из JSON: даты+точность, состав, ветка опыта, интересы+глубина, ритм, бюджет отелей+рёкан, бронирование, формат сопровождения, mobility, заметки). Ниже — сборка программы (Linked Routes + «Создать маршрут»), затем этапы коммуникации/воронка (Stage + таймлайн).

**Стадии:** submit анкеты стадию не двигает (канон prospects-crm-spec); Telegram-уведомление со сводкой профиля и ссылкой на карточку.
