## 1. Безопасность

### 🔴 Критично: Next.js 16.2.0 — advisory на обход middleware

Установлен `next@16.2.0`. Открытые advisory, релевантные именно этой архитектуре:

- *Middleware / Proxy bypass in App Router applications via segment-prefetch routes* (<16.2.6) — **и follow-up к неполному фиксу**;
- *Middleware / Proxy bypass through dynamic route parameter injection* (<16.2.5);
- *Middleware / Proxy redirects can be cache-poisoned* (<16.2.5);
- XSS в App Router с CSP-nonce, cache poisoning в RSC-ответах, SSRF через WebSocket-upgrade, несколько DoS.

Почему это критично именно здесь: `middleware.ts` — **единственный** слой авторизации админки. Ни один `src/app/api/admin/**/route.ts` не проверяет сессию самостоятельно (единственное исключение — `auth/debug`). Успешный обход middleware = полный доступ к админ-API: чтение и запись всех данных Airtable, клиентская база (Prospects с контактами), правка публичных страниц.

**Действие:** обновить Next до ≥ 16.2.6 (`npm i next@^16.2.6`), прогнать build и smoke-тест админки. Это одна из немногих задач, где риск оправдывает немедленный апдейт зависимости.

### 🟠 Высокое: авторизация только в middleware (defense-in-depth отсутствует)

Даже без CVE это архитектурно хрупко: любая ошибка в matcher'е, редирект-правило или будущий рефактор рендеринга открывает админ-API. Матчер сейчас содержит и общий паттерн `'/((?!.well-known/workflow/).*)'`, что делает поведение менее очевидным.

**Действие:** добавить серверный guard в admin-API — хелпер `requireAdminSession()` (`verifySessionToken` из `admin-auth`) и вызов в начале каждого `route.ts` под `/api/admin/**`. Дёшево (одна строка на роут), убирает единственную точку отказа.

### 🟠 Высокое: `POST /api/admin/revalidate` — мутирующий эндпоинт без собственной проверки

Сбрасывает кэш всего сайта. Защищён только middleware (см. выше) и не требует ни секрета, ни тела запроса. При обходе middleware — тривиальный DoS-вектор на кэш (шквал revalidate → ISR-регенерация всех страниц → нагрузка на Airtable).

**Действие:** тот же `requireAdminSession()`; опционально — троттлинг.

### 🟡 Среднее: `GET /api/admin/auth/debug` в проде

Отдаёт имя cookie сессии, факт наличия токена, его длину, e-mail сессии, срок годности и **список имён всех cookie**. Для админа это безобидно, но эндпоинт — готовый разведывательный инструмент, если auth когда-нибудь протечёт. Классика: диагностический роут, забытый после отладки.

**Действие:** удалить, либо закрыть под `NODE_ENV !== 'production'`.

### 🟡 Среднее: Basic-Auth fallback

`ADMIN_BASIC_AUTH_FALLBACK=true` включает вход по логину/паролю из env, сравнение — обычным `===` (не constant-time; для пароля в env практического риска мало, но сам fallback обходит Google-SSO и allowlist e-mail'ов).

**Действие:** убедиться, что в проде переменная не выставлена; при возможности — удалить ветку целиком, раз Google OAuth настроен.

### 🟡 Среднее: остальные уязвимости зависимостей

`npm audit --omit=dev`: 33 уязвимости (1 critical, 17 high). Основная масса — транзитивные зависимости `@workflow/*` и `@xhmikosr/decompress` (critical, path traversal при распаковке архивов — в рантайме сайта не используется, риск теоретический). `npm audit fix` закрывает часть без мажорных бампов.

**Действие:** `npm audit fix`, затем build; остаток — оценить после апдейта Next.

### 🟢 Что сделано хорошо

- **Секреты**: `.env*` в `.gitignore`, в трекинге нет ни одного файла с ключами; все обращения через `process.env`, захардкоженных токенов нет.
- **Сессии**: HMAC-SHA256 (WebCrypto), cookie `httpOnly`, `secure` в проде, TTL 14 дней; PKCE в OAuth-потоке; allowlist e-mail'ов.
- **Инъекции в Airtable formula**: закрыты в новых модулях (`multi-day-builder-storage`, `print-program`, `getRouteStopsByIds` — экранирование кавычек); публичный вход по токену опросника валидируется regex'ом **до** запроса (`TOKEN_PATTERN`) — грамотно.
- **XSS**: `dangerouslySetInnerHTML` — 26 файлов, но везде это `JSON.stringify(schema)` для JSON-LD, пользовательский ввод в HTML не попадает.
- **Формы**: reCAPTCHA v3 + honeypot + rate-limit на `/api/profile`; тихий отказ ботам вместо ошибки.
- **Публичный `/api/revalidate`**: гейт по секрету + allowlist тегов — правильно спроектировано.
- **Админка**: `X-Robots-Tag: noindex`, `Cache-Control: private, no-store`.

### Не проверено

Права Airtable-токена (scope: чтение/запись всей базы?), политика Vercel-переменных, ротация `ADMIN_AUTH_SECRET`, реальный трафик на предмет brute-force.

## 2. Качество кода

### Сильные стороны

- **Типизация**: `: any` / `as any` — **0 вхождений** на 39k строк. `tsc --noEmit` чистый.
- **Лишний шум**: TODO/FIXME/HACK — 0; `console.log` — 6, все осмысленные (логи воронки и антибота); `eslint-disable` — 5, все с обоснованием в комментарии.
- **Кэш-дисциплина**: единая система тегов (`airtable:routes|pois|resources|events`), `unstable_cache` + `revalidateTag(tag,'max')` во всех мутациях — редко встречается такая последовательность.
- **Устойчивость данных**: серверные предохранители сохранения конструктора (shrink/demote/sync-conflict + автобэкап `Program Backup`) — уровень, нетипичный для проекта такого размера.
- **Комментарии**: объясняют *почему*, часто со ссылкой на инцидент. Это заметно выше среднего и реально работает как передача контекста между агентами.

### Слабые места

**🟠 `MultiDayBuilderWorkspace.tsx` — 3094 строки.** Главный технический долг. Внутри: DayCard, диалог идентичности, пикеры POI/отелей/переезда, boot-логика, черновики, синхронизаторы (рейсы, даты, старты, якорь переезда), ~15 обработчиков. Файл рос органически всю неделю и стал местом, где легко посадить регрессию (что и происходило). Естественные швы: `DayCard` + `TransferSection` + `RouteIdentityDialog` в отдельные компоненты, синхронизаторы (`syncFlightItemTitles`, `syncStartLocationChain`, `syncDayTitleDates`) — в `src/lib/multi-day-builder-sync.ts` (чистые функции, тестируемые без React).

**🟠 `japantravel-events.ts` — 1467 строк** и **сломанный парсер индекса** (известно с 2026-07-03, зафиксировано в CLAUDE.md §13): `en.japantravel.com/events` больше не отдаёт JSON-LD в индексе, импортёр стабильно находит 0 событий. Пайплайн фактически мёртв, но код и скрипты живут. Решение: либо чинить парсер (hydration payload), либо честно пометить пайплайн как приостановленный.

**🟡 Крупные файлы второго эшелона**: `AdminResourcesWorkspace.tsx` (1612), `TouristProfileForm.tsx` (1266), `AdminOperationsConsole.tsx` (1085), `resources.ts` (1051). Пока управляемы, но растут.

**🟡 Тесты**: единственный файл `tests/multiday-locale-invariants.mjs`, в `package.json` нет скрипта `test` — то есть он не запускается ни локально, ни в CI. Для проекта, где повторяются баги «даты/пояс/пустое состояние», отсутствие тестов на чистые функции — самый дешёвый из неиспользованных рычагов. Первые кандидаты: `reconcileMultiDayRoute` (ресайз дней), `syncStartLocationChain` (эстафета), `resolveOvernights`/`getRouteStops` (счёт ночей), `slugify`.

**🟡 Дубли**: логика Airtable-фетча (пагинация, retry, экранирование формул) переписана как минимум в трёх местах — `airtable.ts`, `multi-day-builder-storage.ts`, `prospects.ts`, `print-program.ts`. Просится общий `airtable-client.ts`.

**🟡 Заголовки безопасности**: в `next.config.ts` нет `headers()` — ни CSP, ни `X-Frame-Options`, ни HSTS. Для сайта без пользовательского контента риск невелик, но это дешёвая гигиена.
