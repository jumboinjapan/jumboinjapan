# Промпт для Fable 5 — полный аудит JumboInJapan.com

_Подготовлено: 2026-07-02. Версия 2 — оптимизирована по стоимости: Fable 5 подключается только к тяжёлым структурным вопросам, всё механическое делает Sonnet/Opus._

## Почему промпт разбит на два этапа

Fable 5 — дорогая модель ($10/млн входных токенов, $50/млн выходных — в разы дороже Sonnet/Opus), заточенная под глубокое рассуждение поверх огромного контекста. Полный «прочитай весь репозиторий + обойди сайт в 3 разрешениях + напиши отчёт» одним промптом означает, что Fable тратит бюджет на механическую работу (чтение файлов подряд, скриншоты, построчная проверка метатегов), с которой прекрасно справится модель на порядок дешевле.

Поэтому промпт разделён:

- **Этап A — подготовка (Sonnet или Opus, дёшево).** Инвентаризация, запуск lint/tsc, скриншоты сайта, выгрузка метаданных по страницам, список полей Airtable, которые реально используются в коде. Результат — компактные структурированные артефакты, а не сырые дампы.
- **Этап B — структурный анализ (Fable 5, дорого — только это).** Fable получает уже подготовленные артефакты Этапа A и решает только те задачи, где реально нужно удерживать в голове весь проект целиком: архитектурная консистентность, целостность данных между слоями, риски масштабирования, приоритизация. Fable **не** перечитывает то, что уже собрал Этап A, и не занимается тем, что не требует кросс-файлового рассуждения.

Ниже — оба промпта. Сначала прогоните Этап A, вставьте его результат в Этап B и передайте Fable 5.

## ЭТАП A — подготовка (запускать на Sonnet/Opus, не на Fable)

## --- НАЧАЛО ПРОМПТА A ---

Ты готовишь входные данные для отдельного глубокого архитектурного аудита. Твоя задача — не анализировать и не делать выводов, а **собрать и компактно оформить факты**. Экономь токены: не вставляй содержимое файлов целиком, если хватает выдержки или сводки.

Выполни и оформи результат по пунктам:

1. **Карта репозитория.** Дерево `src/app`, `src/components`, `src/lib`, `src/data`, `scripts` с одной строкой описания на файл/модуль (не весь код — только назначение). Отметь файлы длиннее 400 строк отдельным списком — это кандидаты на структурный разбор Fable.

2. **Статический анализ.** Запусти `npx tsc --noEmit` и `npm run lint`. Приложи сырой вывод как есть (если пусто — так и напиши). Не пытайся анализировать причины — просто зафиксируй факты.

3. **Метаданные и schema.org по страницам.** Для каждого типа страницы (главная, `/city-tour/*`, `/intercity/*`, `/multi-day`, `/resources/*`, `/contact`) вытащи блок `generateMetadata`/статичный metadata и весь JSON-LD, который рендерится на странице. Собери в таблицу: страница → title/description → canonical → какие `@type` schema.org есть. Без интерпретации, просто факты рядом друг с другом.

4. **Скриншоты живого сайта.** Пройдись по тем же страницам на 390px (моб.), 768px (планшет), 1440px (десктоп). К каждому скриншоту — одна строка: что видно, есть ли что-то визуально сломанное (наезжающий текст, обрезанные картинки, нечитаемый контраст). Не давай оценок дизайна — только фактические поломки, если они есть.

5. **Airtable-поверхность в коде.** Через grep по `src/lib` собери: какие имена таблиц (`*_TABLE_NAME`) и какие имена полей Airtable реально упоминаются в коде. Отдельно отметь модули, которые читают Airtable в обход общего `resources.ts` (если такие есть) — это сигнал возможного рассинхрона.

6. **Пайплайн Japan Travel.** Запусти `npm run import:japantravel-events -- --pages 1 --limit 5 --dry-run`. Приложи вывод. Не интерпретируй.

7. **a11y-чеклист по факту, не по мнению.** По тем же страницам: есть ли `alt` у всех `<img>`/`next/image`, есть ли видимый `:focus` на интерактивных элементах, минимальная высота тач-таргетов на мобильной версии. Просто да/нет по каждому пункту на каждой странице.

Отдай результат одним документом, разбитым на эти 7 разделов, без общих выводов и рекомендаций — их сделает следующий этап.

## ЭТАП B — структурный анализ (передать Fable 5 вместе с результатом Этапа A)

## --- НАЧАЛО ПРОМПТА B ---

### Роль

Ты — архитектурный аудитор JumboInJapan.com. Тебе НЕ нужно самостоятельно перечитывать весь репозиторий, обходить сайт по страницам или запускать lint/tsc — это уже сделано на подготовительном этапе, результат приложен ниже. Твоя работа начинается там, где заканчивается механический сбор фактов: **синтез, поиск структурных проблем, которые видны только при удержании всего проекта в голове одновременно, и приоритизация.**

Читай отдельные файлы из репозитория (если есть доступ) только тогда, когда конкретный структурный вопрос требует это подтвердить — не читай построчно всё подряд.

### Контекст проекта

JumboInJapan — русскоязычный сайт частного гида по Японии на Next.js (App Router, ^16.2.0) + React 19 + TypeScript + Tailwind 4, с Airtable как основным хранилищем данных и Vercel как хостингом. Продукт должен ощущаться как личный сервис гида — не generic travel-портал. Не предлагай решений, которые обезличивают контент.

Полный операционный контекст — в `CLAUDE.md` в корне репозитория (архитектура, зоны риска, правила эскалации) и `docs/multi-day-route-builder-spec.md` (конструктор маршрутов). Прочитай их, если есть доступ к репозиторию — это единственное, что стоит прочитать полностью перед началом.

### Что именно входит в твою зону ответственности (дорогая, штучная работа)

Фокусируйся только на вопросах, которые требуют кросс-файлового или кросс-слойного рассуждения:

1. **Архитектурная консистентность.** Например: рестораны/услуги/события идут через Airtable по Resource ID-джойну, а отели исторически шли через легаси-паттерн с локальным файлом (сейчас частично исправлено добавлением таблицы `Resource Hotel Partner Links` — проверь, действительно ли новый паттерн выдержан во всех местах, где раньше был легаси-путь). Найди другие места, где похожие сущности реализованы по-разному без причины.

2. **Целостность данных между слоями.** Где то, что лежит в Airtable, расходится с тем, что реально видит пользователь на сайте — из-за кэширования, fallback-логики на seed-данные, или несинхронизированных полей. Используй раздел «Airtable-поверхность в коде» и «Скриншоты» из Этапа A как отправную точку, но делай вывод только если это системная проблема, а не единичный случай отсутствующих данных.

3. **Архитектура производительности.** Не «эта картинка не оптимизирована» (это Этап A уже мог бы заметить), а системные решения: правильно ли расставлена граница server/client компонентов, нет ли повторных некэшированных обращений к Airtable на одной странице, оправдан ли `dynamic = 'force-dynamic'` там, где он стоит.

4. **Архитектура безопасности.** Модель авторизации админки (`ADMIN_AUTH_SECRET`, `ADMIN_ALLOWED_EMAILS`, basic-auth fallback) — нет ли путей её обойти; риски случайной записи в прод через импортёр Japan Travel, если dry-run когда-нибудь перестанет быть поведением по умолчанию.

5. **Системные SEO/schema-риски.** Не построчная проверка одной страницы (это в таблице Этапа A), а: дублирует ли global layout schema то, что уже задают конкретные страницы; логически ли согласована стратегия canonical при множестве похожих маршрутов (`/intercity/*`, `/city-tour/*`).

6. **Технический долг и приоритизация.** На основе файлов длиннее 400 строк (список из Этапа A) и известных болевых точек (см. ниже) — что реально стоит рефакторить в первую очередь, а что можно оставить.

### Что НЕ входит в твою зону — не трать на это токены

Это либо уже сделано Этапом A, либо не требует твоего уровня модели — такое лучше поручить Sonnet/Opus отдельным заходом:

- Построчный разбор вывода lint/tsc и написание конкретных фиксов — это механическая работа для Sonnet.
- Визуальный QA каждой страницы по скриншотам — факты уже собраны в Этапе A, тебе нужны только системные паттерны, если они там есть.
- Редактура русской копии (тон, стиль) — это отдельная задача для Sonnet с учётом голоса бренда, не архитектурный вопрос.
- Проверка alt-текстов/контраста постранично — уже в чеклисте Этапа A.
- Мелкие point-фиксы в одном файле без последствий для других модулей.

Если в процессе работы ты видишь такую мелкую находку — зафиксируй её одной строкой в отдельном разделе отчёта «Делегировать Sonnet/Opus», не разворачивай в отдельный анализ.

### Жёсткие ограничения
