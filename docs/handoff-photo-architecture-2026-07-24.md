## 0. Миссия (одним абзацем)

Система хранения фото сайта jumboinjapan.com работоспособна, но **непредсказуема при изменениях**: путь к одной и той же картинке задаётся в двух местах сразу (код + Airtable), приоритет молчаливо у Airtable, целостность ссылок ничем не проверяется, файлы названы и разложены без правил, копятся дубли и сироты.

**Мандат.** Аудит текущего состояния сделан (ниже, §1–3, §6). Владелец НЕ хочет, чтобы «идеал» диктовали снаружи — **ты сам проектируешь целевую архитектуру хранения фото под текущий стек** (Next 16 App Router + Airtable + Vercel Image Optimization + опционально Dropbox), обосновываешь её, согласуешь с владельцем и приводишь систему к ней. Мои §4/§5 ниже — это НЕ готовое решение, а рамка ограничений и перечень открытых развилок, которые ты должен закрыть своим дизайном. Считай §4 техзаданием на проектирование, а не проектом.

Это НЕ срочный визуальный фикс. Это рефакторинг слоя ассетов. Сначала — предложенный дизайн (RFC на 1–2 страницы) + план, согласование с владельцем; потом исполнение через ветку + preview, с проверками, малыми шагами.

## 2. Текущая архитектура — карта источников правды

Путь к изображению в разных местах берётся по-разному. Это ядро проблемы.

| Поверхность | Откуда путь | Файл / приоритет |
|---|---|---|
| **Hero intercity** (Фудзи, Никко, Эносима, Хаконе, Кёто-1/2, Нара, Удзи, Осака, Химедзи, Каназава, Камакура) | **захардкожен в коде** | `src/app/intercity/<slug>/page.tsx` → `<PageHero image="/tours/…">`. Airtable сюда не вмешивается. |
| **Карточка тура на `/intercity`** | захардкожен в коде | `src/app/intercity/page.tsx` (инлайн-массив `items[]`, поле `image`) |
| **OG/JSON-LD intercity** | из `tours[]` | `src/data/tours.ts` (`image`) |
| **Остановки city-tour** (day-one / day-two / hidden-spots) | **Airtable поверх кода** | код-fallback: инлайн `stops[]` в `page.tsx` (`photo:`,`alt:`); override: Airtable Route Stops `Photo Path`/`Photo Alt`. **Airtable побеждает.** |
| **POI-остановки intercity** (таймлайн на странице тура) | Airtable + fallback | `src/lib/intercity-pois.ts` (`photoPath`), Airtable Route Stops `Photo Path` |
| **Hero многодневных маршрутов** | **из Airtable** | Routes `Hero Image Path` → `route.heroImagePath` (`src/app/multi-day/[slug]/page.tsx:130`, `multi-day/page.tsx:85`) |
| **Hero транспортных под-страниц** (public/private/charter) | захардкожен | `src/app/*/public|private|charter/page.tsx:22` (`heroImage=`) |
| **Карусель галереи** | код | `src/components/sections/ImageCarousel.tsx` (`IMAGE_ALT_BY_PATH`) |

### Механика override (city-tour) — ключевой файл
`src/lib/city-tour-overrides.ts` → `applyCityTourStopOverrides(baseStops, airtableStops)`:
- сопоставляет кодовую остановку с записью Airtable по `POI Name Snapshot` **или** `Stop Title Override` == `stop.title`;
- при совпадении **перекрывает**: `title ← titleOverride || …`, `text ← descriptionOverride || …`, `photo ← record.photoPath || stop.photo`, `alt ← record.photoAlt || stop.alt`;
- т.е. если у остановки в Airtable заполнен `Photo Path` — кодовый `photo:` **мёртв**.

Страница дергает это так (пример `src/app/city-tour/day-one/page.tsx`):
```
export const revalidate = 3600 // ISR, тег 'airtable:routes', сброс через /api/revalidate на admin-write
const stops: CityTourStop[] = [ … ]                       // fallback
const airtableStops = await getIntercityRouteStopsCached('city-tour/day-one')
const sortedStops = applyCityTourStopOverrides(stops, airtableStops)  // Airtable > код
```

### Airtable — что реально хранится
`src/lib/airtable.ts`, тип `AirtableRouteStop`:
- `photoPath ← fields['Photo Path']` (строка, напр. `/tours/city-tour-day-two/shibuya.jpg`)  *(airtable.ts:453)*
- `photoAlt  ← fields['Photo Alt']`  *(airtable.ts:454)*
- Routes-таблица: `Hero Image Path` (для многодневных).

**Т.е. Airtable уже хранит только ссылку-путь, а не сам файл — это соответствует цели.** Файл всегда в `public/`.

## 3. Дефекты (с доказательствами)

1. **Двойной источник правды → тихий дрейф.** Для остановок city-tour путь есть и в коде (`photo:`), и в Airtable (`Photo Path`). Правка кода — no-op, пока в Airtable есть значение. Отладить «на глаз» невозможно: коммит проходит, а на лайве старое.
   - **Живой кейс (Сибуя, 2026-07-24):** код после правки указывал на `hachiko-shibuya.jpg`, а Airtable — на `/tours/city-tour-day-two/shibuya.jpg`. На лайве грузился Airtable-путь; замена кода не дала эффекта. Подтверждено инспекцией DOM боевого билда.
   - В спеке это прямо названо **«урок Мэйдзи/Сибаматы»** — значит это уже случалось. Сибуя — рецидив того же класса бага.

2. **Нет контроля целостности ссылок.** `Photo Path` — свободная строка. Ничто не гарантирует, что файл существует в `public/`. Впишешь путь без файла → битая картинка, без ошибки на сборке. И наоборот — файлы-сироты в `public/`, на которые никто не ссылается.

3. **Хаос в размещении/именовании.** `shibuya.jpg` лежит в `public/tours/city-tour-day-two/`, а показывается на **day-one**. Деление day-one/day-two условное. Именование непоследовательно (`shibuya-crossing-night.jpg`, `shibuya.jpg`, `hachiko-shibuya.jpg`, `fuji-a.jpg` vs `fuji-1.jpg` vs `nara-1.jpg`).

4. **Дубли и сироты.** По одному объекту «Сибуя» в репо сейчас **три файла** (см. §6). `public/tours/uji/` содержит и `fuji-*`, и `uji-*` (проверить). Нет инструмента, чтобы увидеть, что используется, а что мусор.

5. **Кросс-подстановка заглушек.** `src/app/city-tour/takao/page.tsx` и `mitake/page.tsx` используют `shibuya-crossing-night.jpg` как фото **горной вершины** (alt «вид с вершины Такао/Митакэ»). Контент не соответствует.

6. **`Photo Path` не редактируется из админки.** `grep` по `src/app/admin`, `src/app/api/admin` — пусто. Единственная точка правки — руками в Airtable, без валидации.

7. **Dropbox-мастер-архив спроектирован, но не подключён.** `dropbox-photo-archive-spec-2026-07-21.md` описывает хорошую схему (`site/` зеркалит `public/` 1:1, оригиналы отдельно), но коннектор не авторизован → пока только `public/`.

## 4. Рамка проектирования (ТЗ, не готовое решение)

Идеал определяешь ты, под текущий стек. Ниже — **жёсткие принципы владельца** (не обсуждаются) и **открытые развилки**, которые твой дизайн должен закрыть и обосновать.

### 4.1 Жёсткие принципы (ограничения, не трогать)
- **Все файлы фото — в одном месте одного репо** (`public/…`). Один стор. Не размазывать по нескольким источникам в рантайме.
- **Airtable хранит только ссылку/путь, никогда не сам файл** (байты картинки живут в репо).
- **Никакого тихого дрейфа.** На каждую картинку — один очевидный, предсказуемый способ её поменять. «Поменял в одном месте — коммит прошёл — на лайве старое» повторяться не должно.
- **Оставаться на текущем стеке.** Next 16 App Router + Airtable + Vercel Image Optimization + ISR (`tag 'airtable:routes'`). Реплатформинг предлагать только с очень сильным обоснованием и отдельным согласием владельца.
- Соблюдать ограничения §7 (нет push из Cowork, Airtable-safety, preview для общего слоя, sandbox не собирает `next build`).

### 4.2 Открытые развилки — реши и обоснуй в RFC
1. **Единый источник правды на каждый тип поверхности.** Где должен жить авторитетный путь для: hero intercity, hero многодневных, остановок city-tour, POI-остановок intercity, карточек/OG? Код vs Airtable. Взвесь: кто редактирует (владелец через Airtable/админку vs разработчик через git), версионирование, безопасность preview, единообразие. Сейчас — мешанина (см. §2); нужна принятая, обоснованная схема.
2. **Как убрать дублирование код↔Airtable для остановок**, не потеряв безопасный fallback (пустой Airtable не должен ронять страницу в «нет фото»).
3. **Механизм контроля целостности.** Как гарантировать, что каждый путь из Airtable/кода резолвится в существующий файл `public/`, и как ловить сироты. Варианты на выбор: скрипт + CI-гейт, типизированный манифест ассетов (кодоген из списка файлов), рантайм-гвард, их комбинация. Обоснуй.
4. **Схема именования и раскладки `public/`** (и надо ли зеркалить структуру под будущий Dropbox `site/`). Учти §4.1 и спеку.
5. **Кэш оптимизатора картинок Vercel** кэширует по URL: замена байтов по тому же пути может «висеть» старой. Реши как (контент-хэш в имени? версионирование пути? осознанный сброс?). Это влияет на схему именования (п.4).
6. **Роль Dropbox-архива** (`dropbox-photo-archive-spec-2026-07-21.md`): принять сейчас / позже / отклонить. Если принять — как соотносится с репо-стором из §4.1 (архив оригиналов ≠ рантайм-источник; `public/` остаётся единственным, что читает сайт).
7. **Точка редактирования для владельца** (не-технический поток): нужна ли админ-поверхность правки фото остановки, или достаточно задокументированного процесса.

Выдай это как короткий RFC (текущее → предлагаемое → миграция → риски), согласуй с владельцем, потом исполняй.

## 5. Кандидатная последовательность (адаптируй под свой RFC)

Это НЕ обязательный план — финальный порядок задаёт твой дизайн (§4). Но логика «сначала видимость, потом единый источник правды, потом чистка, потом Dropbox» проверена и, скорее всего, останется. Фаза 1 полезна при любом дизайне — делай её первой.

**Фаза 1 — Валидатор (дешево, максимально полезно, без риска).**
- Написать `scripts/check-image-refs.mjs`: тянет Route Stops (`Photo Path`,`Photo Alt`) и Routes (`Hero Image Path`) из Airtable (env `AIRTABLE_TOKEN`,`AIRTABLE_BASE_ID`), собирает все кодовые пути (`grep`/AST по `image=`,`photo:`,`heroImage`,`/tours/…`), сверяет с `fs` по `public/`. Вывод: (1) ссылки без файла, (2) файлы-сироты, (3) дубли по хешу, (4) нарушения мин.размера.
- Прогнать, приложить отчёт. **Это сразу даст полную карту текущего бардака** — до любых правок.
- Добавить как `npm run check:images` и в CI (или хотя бы в `docs`-чек).

**Фаза 2 — Единый источник правды для остановок.**
- Убедиться, что для каждой остановки city-tour, у которой в Airtable есть `Photo Path`, кодовый `photo:` избыточен. Затем убрать `photo:`/`alt:` из инлайн-`stops[]` (city-tour day-one/two/hidden-spots), оставив тексты. Обновить тип `CityTourStop`/`CityTourStopLike`, чтобы `photo` приходил только из Airtable (или из явного «seed»-слоя, не из страницы).
- Для остановок БЕЗ записи в Airtable — либо завести запись (предпочтительно, единый источник), либо оставить явный минимальный seed. **Не** оставлять два живых пути к одному объекту.
- Проверка: DOM боевого/preview билда для day-one/two/hidden-spots и всех intercity — картинки не изменились (или изменились осознанно).

**Фаза 3 — Hero-решение (по §8-1).** Привести все hero к выбранному единому авторитету.

**Фаза 4 — Реорг и дедуп `public/`.**
- Переименовать/переложить по правилам §4D, зеркаля будущий `site/`. Каждое переименование = обновить и код, и Airtable `Photo Path` (в одном согласованном шаге; помнить про сброс ISR-кэша `/api/revalidate`).
- Удалить сироты и дубли (из отчёта Фазы 1). Осторожно: «сирота в коде» может быть жив через Airtable — сверяться с валидатором, а не с одним `grep`.
- Починить заглушки Такао/Митакэ (дать реальные фото вершин или явные корректные ссылки).

**Фаза 5 — Dropbox-архив.** По подключении коннектора (владелец, §8-3): создать структуру, залить снимок текущего `public/` в `site/`, задокументировать поток «оригинал → site → public → commit + Airtable». Опц.: перенос оригиналов.

**Фаза 6 — Точка редактирования.** Мини-админка «фото остановки»: меняет `Photo Path`+`Photo Alt` в Airtable, проверяет наличие файла, (опц.) тянет из Dropbox. Сбрасывает кэш.

## 7. Ограничения и правила проекта (соблюдать)

Из `CLAUDE.md` репо и практики сессий:

- **Cowork не может `git push`** (нет кред) и git-операции в примонтированной папке падают на `.lock` (`Operation not permitted`). Коммить локально не выйдет из Cowork штатно; итоговый пуш делает владелец или Lloyd (нативно). Планируй так, чтобы результат ты **готовил патчами**, а пуш оставлял владельцу.
- **Airtable-запись:** dry-run по умолчанию, явный `--write`. Не вводить разрушающих удалений. `Photo Path` — правка существующих полей, безопасно, но батч-переименования делать транзакционно и логировать «было → стало». Не переименовывать поля/таблицы Airtable.
- **ISR-кэш:** прямые правки Airtable не сбрасывают кэш сайта до часа. После правок `Photo Path` дергать `POST /api/admin/revalidate` (кнопка «Обновить кэш сайта» в Route Texts) или `/api/revalidate` (тег `airtable:routes`).
- **Кэш оптимизатора картинок Vercel** кэширует по URL: если оставить прежний путь и заменить байты — старое может висеть. **Предпочитать новый путь/имя файла при замене содержимого** (свежий URL не кэширован) — это ещё одна причина навести именование.
- **Превью-деплой обязателен** для изменений общего слоя всех страниц. Слой резолва картинок/тип `CityTourStop`/override трогает много страниц → катить через ветку + Vercel preview, проверка владельцем, потом main.
- **Sandbox не собирает `next build`** (только `tsc`/`eslint`) — поведенческие баги (как этот) доезжают до прода. Обязателен **adversarial-прогон** и проверка на preview живым DOM.
- Не менять канонические URL, redirects, схему Airtable-полей, позиционирование. Никаких секретов в коммитах.
- Не путать hero intercity (код) с остановками (Airtable) — разные слои (см. §2).
