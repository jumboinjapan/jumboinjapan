### Конструктор туров (`/admin/multi-day`)

- **Серверные предохранители сохранения** (`saveMultiDayBuilderRoute`): `SHRINK_BLOCKED` (пустое/усохшее поверх полного), `DEMOTE_BLOCKED` (снятие с публикации побочным эффектом), `SYNC_CONFLICT` (правки из другой вкладки), `SLUG_TAKEN`. Override — повторным «Сохранить» в течение 15 с; у SYNC_CONFLICT и SLUG_TAKEN override нет. Перед каждой перезаписью — автоснапшот программы в `Routes.'Program Backup'`. Детали в CLAUDE.md §13.
- **Идентичность маршрута**: slug больше не регенерируется из названия (это плодило программы-дубли — в базе жили «Классическая Япония» и «Япония — Первое касание» одновременно). Slug — явное поле в Параметрах; смена slug у сохранённого тура = серверный **rename** той же записи (`saveOptions.previousSlug`), а не создание новой.
- **Диалог создания/дублирования**: «Новый маршрут» и «Дублировать программу» открывают окно, где RU/EN-название и slug задаются ДО создания. Ни один путь больше не порождает программу с самоназначенной идентичностью.
- **Ресайз дней по правилам владельца**: больше дней → пустые экскурсионные встают перед днём вылета; меньше → удаляются дни перед вылетом. Программа существующих дней не гибнет (раньше сопоставление по индексу убивало контент). Тип дня («самостоятельно») больше не сбрасывается при сохранении.
- **Перестановка дней** стрелками ↑↓ (прилёт и вылет неподвижны).
- **Эстафета стартов**: ночёвка — данные, старт — производная. Ночёвка прописывается вперёд до дня вылета; «Старт» дня вычисляется из ночёвки предыдущего и не редактируется руками (кроме дня прилёта — там аэропорт).
- **Вариации переезда дня**: до 3 вариантов на день (ЖД / Авиа / Частный транспорт / Общественный транспорт / Заказной транспорт). Новые поля в Transport Segments: `Service Number`, `Departure Mode`, `Departure With Guide` (чекбокс — гид ортогонален способу выезда), `Recommended Departure Time`, `Guest Comments` (публичный текст). Позицию переезда в дне задаёт блок «Переезд» в списке точек — двигается стрелками, не прибит к низу дня.
- **Политика черновиков**: кэш несохранённых правок НИКОГДА не применяется молча — только явным выбором в баннере. Черновик-скелет (≤2 блоков) при живой серверной программе (≥5) самоудаляется.

## 2-история. Бывшая «главная нерешённая загадка»: admin guard 401-ит в проде, но работает на превью

**Симптом.** `requireAdminSession()` (`src/lib/admin-guard.ts`), добавленный во все `/api/admin/**`-роуты, в **проде** отклоняет ВСЕ запросы: конструктор и Route Stops показывают пустые списки, хотя владелец залогинен и middleware его пропускает. На **preview-деплое той же ветки** тот же код работает штатно (проверено владельцем: маршруты видны, сохранение проходит).

**Что уже исключено:**
- Не данные: маршруты в Airtable на месте (проверено через MCP).
- Не первая версия guard'а: v1 читал cookie через `cookies()` из `next/headers` — падал. v2 читает `request.cookies`, как рабочий middleware — на превью ок, в проде всё равно падает.
- Не Airtable-креды и не сборка: билд проходит, публичный сайт работает.

**Текущее состояние:** guard **снят с main** (коммит `9c5a8f0`). Файл `src/lib/admin-guard.ts` в репозитории остался, вызовы из роутов убраны. Рабочая ветка с включённым guard'ом: `security/admin-guard` (коммит `212706f`).

**Как диагностировать (следующий шаг).** Guard v2 при отказе пишет в лог причину:

```
[admin-guard] denied: /api/admin/multi-day/route no session cookie
[admin-guard] denied: /api/admin/multi-day/route invalid/expired session token
[admin-guard] session verify failed: <причина>
```

Нужно поднять деплой с guard'ом (ветка `security/admin-guard` или временный revert-of-revert), воспроизвести в проде и посмотреть **Vercel → Logs**. Строка скажет, чего именно не видит guard. Гипотезы, которые стоит проверить в этом порядке:

1. Cookie до route handler в проде не доезжает (домен/`www`-редирект/прокси-слой), а до middleware доезжает.
2. `ADMIN_AUTH_SECRET` в рантайме функций отличается от того, которым подписана cookie (например, значение переопределено на уровне Team/Shared-переменных, а не проекта).
3. Разница окружений Production vs Preview в самой Vercel (Deployment Protection, разные регионы функций).

**Важно:** MCP-коннектор Vercel в этой сессии отдавал **403** на `list_deployments`, `get_runtime_logs`, `get_runtime_errors` — логи пришлось бы смотреть владельцу вручную. Если у следующего агента коннектор жив, это решается за минуту.

**Риск от отсутствия guard'а сейчас — умеренный:** обновление Next закрыло известные CVE на обход middleware, а middleware по-прежнему защищает и `/admin`, и `/api/admin`. Guard был вторым рубежом, а не единственным.

## 3. Что важно знать про рабочий процесс (выучено болью)

- **Превью обязателен** для всего, что трогает авторизацию, middleware, env-контракт или общий слой всех роутов. Песочница Cowork не собирает Next — `tsc` чистый, а прод падает. Дважды сломали админку владельцу.
- **Если прод сломан — сначала откат (3 минуты), потом расследование.** Не держать владельца в сломанном состоянии ради диагностики.
- **Владелец не обязан лазить в DevTools.** Просьбы вида «посмотри статус запроса в Network» вызывают справедливое раздражение. Диагностику надо строить на логах, которые агент читает сам, либо на одном скриншоте.
- **Не запускать `npm install`/`npm audit fix` из песочницы**: node_modules принадлежит macOS-машине владельца, Linux-установка их ломает (пришлось просить `rm -rf node_modules && npm install`).
- Проверка из песочницы = `npx tsc --noEmit` (+ `npx eslint`). `npm run build` недоступен.
- Airtable-коннектор **не умеет добавлять опции в существующие select-поля** — владелец добавляет их руками (так было с `private` в Departure Mode).

## Postscript 2026-07-14: инцидент «открытая админка» и разгадка guard-401

**Факт:** с момента перехода на Next 16 прод-админка (`/admin` и весь `/api/admin/**`, включая клиентскую базу) была публично доступна. Анонимный запрос получал 200. Причина: **Next 16 переименовал `middleware.ts` в `proxy.ts` и молча игнорирует старый файл при сборке** — ни ошибки, ни предупреждения в рантайме ([nextjs.org/docs/messages/middleware-to-proxy](https://nextjs.org/docs/messages/middleware-to-proxy)).

**Почему это выглядело как «guard сломан в проде»:** middleware не работал нигде, логин никто не спрашивал, владелец пользовался админкой без сессии. Guard стал первой настоящей проверкой — и честно 401-ил всех. «Работает на превью» из прошлой сессии объяснялось живой сессией на том превью-домене после ручного логина.

**Как нашли:** временный роут `/api/admin/auth/diagnose` (только булевы флаги + hash-отпечаток секрета) показал на превью `hasSessionCookie:false` при успешном ответе защищённого роута → middleware не исполняется → анонимный curl прода подтвердил 200 на `/api/admin/**`.

**Фикс (в main):**
- `f56199c` — `middleware.ts` → `proxy.ts` (функция `proxy`, `runtime` из config удалён — proxy всегда Node.js, опция бросает ошибку).
- `22f8f87` — файл перенесён в `src/proxy.ts`: **при src-структуре Next подхватывает proxy только из `src/`**, корневой файл тоже игнорируется (проверено экспериментально на превью).
- Проверка прода после мержа: `/admin` → 307 на логин, `/api/admin/**` → 401, публичные страницы и легаси-редиректы целы, `ab-hakone` снова ставится.

**Уроки:**
- При любом major-обновлении Next сверять миграционные заметки по каждой файловой конвенции: `tsc`/`eslint`/успешная сборка НЕ ловят молча отключившийся middleware.
- Периодическая проверка «злоумышленника»: анонимный `curl /admin` и `/api/admin/...` должен возвращать 307/401. Стоит добавить в чек-лист деплоя.
- Vercel MCP-коннектору выдан доступ только к проектам `mohan*` — добавить `jumboinjapan` в Project Access, чтобы агент читал логи сам.

**Follow-up:**
1. Удалить `/api/admin/auth/diagnose` (временный, сейчас за proxy).
2. Вернуть guard `requireAdminSession` во все admin-роуты (ветка `security/admin-guard`) — теперь он будет работать: сессии реально существуют. Это восстановит второй рубеж.
3. Ротация `ADMIN_AUTH_SECRET` и `AIRTABLE_TOKEN` — период экспозиции админ-API неизвестен точно (с апгрейда на Next 16); данные могли читаться и писаться анонимно. Проверить Airtable на подозрительные правки.
