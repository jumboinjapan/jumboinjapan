## 2. Целевая модель

### 2.1 Один авторитет на тип поверхности — по принципу «кто редактирует»

| Поверхность | Авторитет пути | Обоснование |
|---|---|---|
| Фото остановок city-tour | **Airtable** `Route Stops.Photo Path/Alt` — единственный | Контент остановок владелец правит без разработчика; уже де-факто так |
| Фото POI-остановок intercity | **Airtable** (тот же контракт) | Единообразие; кодовые seed Хаконе переносятся в Airtable |
| Hero многодневных | **Airtable** `Routes.Hero Image Path` | Маршруты создаются из админки; уже так задумано |
| Hero статических страниц (intercity/<slug>, city-tour, транспортные) | **Код** (`page.tsx`) | Меняются только вместе с дизайном, через git/preview |
| Карточки, OG, JSON-LD | **Код** (`tours.ts`, `page.tsx`) | То же |

Правило: **у одного изображения ровно один живой источник пути.** Кодовые `photo:`/`alt:` в `stops[]` city-tour страниц удаляются.

### 2.2 Fallback без второго источника правды

Пустой/недоступный Airtable не должен ронять страницы. Вместо ручного дубля в коде — **генерируемый снапшот**: `npm run sync:photo-fallback` пишет `src/data/route-stop-photos.generated.json` из Airtable; страницы используют его только когда живой запрос упал. Файл не редактируется руками (заголовок-предупреждение + проверка в валидаторе), поэтому источником правды не является — это кэш, зафиксированный в git.

### 2.3 Контроль целостности — валидатор в CI (уже написан)

`scripts/check-image-refs.mjs` (готов, прогнан): сверяет код + Airtable + `public/`; битая ссылка = падение CI (`npm run check:images`). Ловит сироты, дубли по хэшу, кросс-папки. Это гарантия «вписал путь без файла — узнаёшь до деплоя, а не по битой картинке».

### 2.4 Канон именования и раскладки `public/`

```
/tours/<scope>/<объект-съёмки>[-vN].jpg
```

- `<scope>` — направление/раздел, не поверхность показа: `city-tour` (одна папка вместо day-one/day-two/hidden-spots — деление по дням условное и уже нарушено), `fuji`, `hakone`, …, `multi-day`.
- `<объект>` — что на кадре, kebab-латиницей: `shibuya-hachiko.jpg`, `harajuku-takeshita.jpg`. Никаких `photo1.jpg`, `image.webp`.
- **Замена содержимого = новое имя** (`-v2`, `-v3`): кэш оптимизатора Vercel ключуется по URL, перезапись байтов по старому пути показывает старый кадр непредсказуемо долго. Перезапись байтов под тем же именем — запрещена.
- Структура 1:1 зеркалит будущий Dropbox `site/` из спеки 2026-07-21.

### 2.5 Dropbox-архив — принять, но как архив, не как рантайм

`public/` остаётся единственным, что читает сайт. Dropbox `site/` = зеркало-снимок + `originals/` для исходников. Подключение требует авторизации коннектора владельцем; не блокирует остальные фазы.

### 2.6 Точка редактирования для владельца

Минимальная админ-поверхность в существующем редакторе остановок: поля Photo Path/Photo Alt с проверкой «файл существует в public/» (список известен серверу на билде) + существующая кнопка сброса кэша. До её появления — задокументированный процесс: файл в `public/` (коммит) → путь в Airtable → «Обновить кэш сайта».

## 3. План миграции (фазы = отдельные проверяемые шаги)

1. **Валидатор + CI** — сделано, добавить `check:images` в package.json и CI. Риск: нет.
2. **OG Такао/Митакэ** — убрать битые ссылки (указать существующие hero). Риск: нет.
3. **Единый источник city-tour остановок** — удалить `photo:`/`alt:` из трёх `page.tsx`, тип обновить, включить generated-fallback. Проверка: DOM на preview — картинки не изменились. Через ветку + Vercel preview.
4. **Сибуя → Хатико** — файл `public/tours/city-tour/shibuya-hachiko.jpg`, Airtable Photo Path/Alt остановки Сибуя, сброс кэша; удалить оба старых кода-дубля.
5. **Чистка** — сироты и дубли из отчёта: в `_to_delete/` (байты не уничтожаем — уедут в Dropbox `originals/`); `fuji-b/c` схлопнуть с `fuji-2/3`.
6. **Реорг папок city-tour** (если одобрен) — транзакционно: переложить файлы + обновить Airtable (лог «было→стало») + код + revalidate.
7. **Такао/Митакэ честные фото** — нужны кадры от владельца; до тех пор — явные нейтральные заглушки без ложных альтов.
8. **Dropbox** (по авторизации коннектора) + **админ-поле фото** (фаза 6) — отдельными заходами.

Исполнение: коммиты в облачном клоне → `git bundle` → владелец/Lloyd пушат; превью Vercel до merge. Airtable — dry-run по умолчанию, запись с логом.

## 4. Риски

- Реорг папок трогает живые URL → строго транзакционно с обновлением Airtable и сбросом кэша; старые URL картинок внешним SEO почти не индексируются (это не страницы), риск низкий.
- Generated-fallback может протухнуть → валидатор сверяет его с Airtable и предупреждает при расхождении (сделаю предупреждение частью `check:images`).
- Удаление сирот необратимо → поэтому только перемещение в `_to_delete/`/архив, не `rm`.
