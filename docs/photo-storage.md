# Как устроено хранение фото

**Канон 2026-07-24** (RFC: docs/rfc-photo-architecture-2026-07-24.md, согласован владельцем).

## Три правила, которые нельзя нарушать

1. **Файлы — только в `public/` этого репо.** Airtable хранит только путь-строку, никогда сам файл. Внешних CDN на страницах нет.
2. **У каждого изображения ровно один источник правды пути** (см. таблицу ниже). Дублировать путь во втором месте запрещено — так возникал тихий дрейф (инциденты «Мэйдзи/Сибаматы», «Сибуя 2026-07-24»: правка кода была no-op, потому что Airtable молча перекрывал).
3. **Замена содержимого = новое имя файла** (`-v2`, `-v3`, …). Кэш оптимизатора картинок Vercel ключуется по URL: перезапись байтов под старым именем показывает старый кадр непредсказуемо долго. Перезаписывать байты под тем же путём запрещено.

## Карта источников правды

| Поверхность | Источник пути | Где именно |
|---|---|---|
| Фото остановок city-tour (day-one/day-two/hidden-spots) | **Airtable** | `Route Stops`.«Photo Path» / «Photo Alt» |
| Фото POI-остановок intercity | **Airtable** (тот же контракт) | `Route Stops`.«Photo Path» (сегодня заполнено только у city-tour; кодовый seed Хаконе в `intercity-pois.ts` — унаследованное исключение, мигрировать при случае) |
| Hero многодневных маршрутов | **Airtable** | `Routes`.«Hero Image Path» (пусто → дефолтная обложка раздела) |
| Hero статических страниц (intercity/<slug>, city-tour/*, транспортные) | **Код** | `page.tsx` → `<PageHero image=…>` |
| Карточки, OG, JSON-LD | **Код** | `src/data/tours.ts`, `page.tsx` metadata |

Кодовых `photo:`/`alt:` в `stops[]` городских страниц больше НЕТ. Fallback при недоступности Airtable — `src/data/route-stop-photos.generated.json`: генерируется `npm run sync:photo-fallback`, руками не редактируется, источником правды не является.

## Раскладка и именование `public/`

```
/tours/<раздел>/<объект-съёмки>[-vN].jpg
```

- `<раздел>` — направление, не поверхность показа: `city-tour` (все городские точки Токио — одна папка), `fuji`, `hakone`, … `multi-day`.
- `<объект-съёмки>` — что на кадре, kebab-латиницей: `shibuya-hachiko.jpg`, `harajuku-takeshita.jpg`. Никаких `photo1.jpg`, `image.webp`, `img_1234.jpg`.
- Структура 1:1 готова зеркалить будущий Dropbox-архив `site/` (спека 2026-07-21; подключение отложено решением владельца 2026-07-24).

## Контроль целостности

`npm run check:images` (`scripts/check-image-refs.mjs`) сверяет файлы `public/` ↔ пути в коде/скриптах/generated-JSON ↔ пути в Airtable. Битая ссылка → exit 1 (гейт для CI). Сироты, дубли по SHA-1 и кросс-папочные ссылки — warning. **Гонять перед каждым коммитом, трогающим фото.** Сирот определять только валидатором, не `grep` по src (ссылка может жить в Airtable).

## Процесс замены/добавления фото остановки (пока нет админ-поля)

1. Положить файл в `public/tours/<раздел>/<объект>.jpg` (новое имя!), закоммитить, задеплоить.
2. Прописать путь в Airtable `Route Stops`.«Photo Path» (+ «Photo Alt») — только после деплоя файла.
3. Нажать «Обновить кэш сайта» в админке (Route Texts) — прямые правки Airtable не сбрасывают ISR (иначе ждать до часа).
4. `npm run sync:photo-fallback` и закоммитить обновлённый снапшот.
5. `npm run check:images` — убедиться: 0 битых, старый файл попал в сироты → удалить его отдельным коммитом (байты — в `_to_delete/photo-archive-*/`).

Порядок «файл раньше пути» обязателен: Airtable читается лайвом немедленно, файл появляется только с деплоем.
